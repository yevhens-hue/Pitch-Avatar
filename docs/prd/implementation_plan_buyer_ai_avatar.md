## 1. Базова архітектура & Схема бази даних (Supabase PostgreSQL)

Для інтеграції Buyer AI Avatar, налаштувань тренування та аналітики результатів необхідно розширити схему бази даних Supabase наступними таблицями:

### 1.0. Розширення таблиці `projects`
Додавання прапорця для ідентифікації проектів, що переведені в режим коучингу.
```sql
ALTER TABLE projects ADD COLUMN is_coach_mode BOOLEAN DEFAULT false;
```

### 1.1. Таблиця `coach_settings`
Зберігає параметри тестування та відображення підказок для конкретного проекту.
```sql
CREATE TABLE coach_settings (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  project_id UUID UNIQUE NOT NULL REFERENCES projects(id) ON DELETE CASCADE,
  show_answer_format VARCHAR(20) DEFAULT 'text' CHECK (show_answer_format IN ('text', 'voice', 'none')),
  evaluate_immediately BOOLEAN DEFAULT true,
  allow_skip BOOLEAN DEFAULT false,
  max_questions INTEGER DEFAULT 5,
  question_delivery VARCHAR(20) DEFAULT 'random' CHECK (question_delivery IN ('random', 'sequential')), -- Порядок питань
  start_from_slide_id VARCHAR(100), -- З якого слайду починається тренування (опціонально, напр. з останнього)
  created_at TIMESTAMP WITH TIME ZONE DEFAULT timezone('utc'::text, now()) NOT NULL
);
```

### 1.2. Таблиця `buyer_scenarios` (Q&A База)
Зберігає пари "Запитання-Відповідь" (відв'язані від конкретного слайду, але можуть містити опціональний `expected_slide_id`).
```sql
CREATE TABLE buyer_scenarios (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  project_id UUID NOT NULL REFERENCES projects(id) ON DELETE CASCADE,
  role_id UUID, -- Посилання на існуючий конструктор ролей (ICP / Use Cases)
  question_text TEXT NOT NULL,
  expected_answer TEXT NOT NULL,
  expected_slide_id VARCHAR(100), -- ОПЦІОНАЛЬНО: ID слайду, який бажано показати
  is_generated BOOLEAN DEFAULT false, -- Чи згенеровано штучно AI
  order_index INTEGER DEFAULT 0, -- Для послідовної видачі питань
  custom_actions JSONB DEFAULT '[]', -- Кастомні інтерактивні кнопки дій
  created_at TIMESTAMP WITH TIME ZONE DEFAULT timezone('utc'::text, now()) NOT NULL
);
```

### 1.3. Таблиця `training_sessions` (Аналітика результатів)
Зберігає результати проходження тренування слухачами-продавцями (інтеграція з HR Onboarding Results).
```sql
CREATE TABLE training_sessions (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  project_id UUID NOT NULL REFERENCES projects(id) ON DELETE CASCADE,
  listener_id UUID REFERENCES listeners(id) ON DELETE SET NULL, -- Хто проходив
  score INTEGER DEFAULT 0 CHECK (score >= 0 AND score <= 100),
  is_train_mode BOOLEAN DEFAULT false, -- Навчання чи тестування
  session_logs JSONB DEFAULT '[]', -- Детальна історія: [{question, user_answer, is_correct, slide_shown}]
  created_at TIMESTAMP WITH TIME ZONE DEFAULT timezone('utc'::text, now()) NOT NULL
);
```

## 2. API Routes (Next.js App Router)

Нам знадобляться 3 нових API ендпоінти:

### 2.1. `POST /api/coach/generate-questions`
*   **Опис**: Генерує стандартні запитання покупця на основі вмісту презентації та наявної **ролі покупця (з конструктора ролей)**.
*   **Логіка**: Зчитує контент проекту + метадані вибраної існуючої ролі (ICP) ➔ Відправляє LLM промпт з цією роллю ➔ Повертає список питань. Це дозволяє Аватару бути проактивним та самостійно ініціювати розмову.

### 2.2. `POST /api/coach/evaluate`
*   **Опис**: Оцінює відповідь продавця в реальному часі (якщо увімкнено `evaluate_immediately`).
*   **Логіка**: Порівнює `user_answer` із `expected_answer` (через LLM-based Cosine Similarity) та перевіряє `slide_shown` з `expected_slide_id`.

### 2.3. `POST /api/coach/save-to-rag`
*   **Опис**: Ручне не-тригерне збереження питань/відповідей у базу знань RAG проекту або як тестові інструкції.
*   **Логіка**: Записує новий чанк у векторний індекс проекту (`knowledge_documents`) для Grounded RAG або створює запис у `buyer_scenarios`.

## 3. Фронтенд-компоненти (React / Zustand / CSS Modules)

### 3.1. Візард налаштування Coach Settings (`/src/components/Coach/CoachSettingsWizard.tsx`)
*   Форма з перемикачами:
    *   Режим показу підказок: `Text`, `Voice`, `None`.
    *   Тумблер: `Evaluate immediately`.
    *   Порядок питань: `Random` (вибрати N з M) або `Sequential` (всі по черзі).
    *   Input: `Number of questions` (для Random режиму).
    *   Вибір ролі: Підключення до існуючого списку Use Cases / ICP ролей.

### 3.2. Тренувальний плеєр (`/src/components/Coach/CoachPlayer.tsx`)
*   **Left Section (Stage)**:
    *   Карусель слайдів з можливістю швидкого перемикання продавцем.
    *   Аватар-Покупець (відео або аудіо з lip-sync), що задає питання.
*   **Bottom Toolbar**:
    *   Кнопка `Show Answer` (показує текст підказки, підсвічує релевантний слайд).
    *   Чек-бокс `Check answer` (миттєве підтвердження).
    *   Кнопка запису голосу/введення тексту відповіді.
*   **Right Section (Chat & Dynamic Actions)**:
    *   Відображення діалогу покупця та продавця.
    *   Кастомні інтерактивні кнопки дій (наприклад, "Offer Discount" або "Schedule Demo"), створені через функціонал *Add Action*.

### 3.3. Train Mode панель керування (`/src/components/Coach/TrainControls.tsx`)
*   Панель адміністратора/експерта ("Тренера") для проходження еталонного сценарію.
*   Тренер відповідає на запитання, коригує логіку LLM у реальному часі та зберігає ідеальну реакцію.
*   Кнопка **"Save to RAG"** та **"Save as Test Instruction"** (ручні збереження).
*   Тренер може опціонально прив'язати відповідь до слайду (частний випадок, але потрібен).

## 5. Технічні питання для затвердження (Technical Questions for Sign-Off)

Для фіналізації архітектури та переходу до фази розробки, просимо затвердити або надати коментарі щодо наступних 5 питань:

### ❓ 5.1. Тип та зв'язок слайдів (`expected_slide_id`)
Згідно з обговоренням, показ слайду — це опціональний (частний) випадок. Запитання відв'язані від конкретних слайдів і є сутністю Knowledge Base.
*   *Питання:* Чи достатньо для MVP залишати `expected_slide_id` необов'язковим текстовим полем, і якщо тренер під час Train Mode його не обрав, то при оцінці продавця система взагалі не зважатиме на те, який слайд зараз відкритий?

### ❓ 5.2. Структура `custom_actions` та розуміння їх AI-аватаром
Під функціоналом *Add Action* ми плануємо зберігати динамічні кнопки у JSONB-полі `custom_actions`.
*   *Питання:* Якою має бути структура кожної дії? Чи достатньо наступного формату:
    ```json
    [
      {
        "action_key": "offer_discount_10",
        "label": "Запропонувати знижку 10%",
        "expected_result": "positive_response",
        "score_modifier": 10
      }
    ]
    ```
*   Чи повинен AI-покупець бути в курсі натискання цієї кнопки (тобто ми передаємо опис дії у контекст LLM, щоб аватар реагував відповідно), чи це просто "кнопка-тригер" для нарахування балів у системі оцінювання?

### ❓ 5.3. Асинхронність оцінювання мовлення продавця (`/api/coach/evaluate`)
Якщо продавець відповідає довгим голосовим повідомленням, процес транскрибації (Whisper) + LLM-оцінка відповіді може тривати від 5 до 10 секунд.
*   *Питання:* Чи робимо ми оцінку повністю асинхронною через статусний флоу (клієнт відправляє аудіо ➔ отримує статус `evaluating` ➔ інтерфейс показує скелетон/лоадер ➔ клієнт опитує сервер про готовність або чекає повідомлення), чи за рахунок жорсткого ліміту на тривалість відповіді (наприклад, до 15 секунд) ми можемо тримати запит синхронним?

### ❓ 5.4. Ізоляція RAG даних навчання від публічного RAG (`/api/coach/save-to-rag`)
При збереженні правильних відповідей у векторну базу RAG у Train Mode:
*   **Питання:** Чи повинні ці тренувальні пари (`питання -> відповідь`) використовуватися публічним аватаром на сайті під час розмови з реальними покупцями, чи вони мають бути суворо ізольовані та використовуватись *виключно* для навчання продавців? Якщо ізольовані, нам потрібно додати мета-тег `is_training_only: true` для векторного пошуку.

### ❓ 5.5. Збереження проміжного стану при `evaluate_immediately = false`
Якщо налаштовано глобальний параметр "оцінювати в кінці, а не відразу":
*   **Питання:** Де ми зберігаємо проміжні відповіді продавця та перемикання слайдів протягом тесту?
    *   *Варіант А:* У стані клієнта (Zustand/React Context) та відправляємо все одним великим пакетом в кінці. (Легко розробляти, але при випадковому оновленні сторінки хід тесту втрачається).
    *   *Варіант В:* Записуємо в БД у таблицю `training_sessions.session_logs` як чернетку (`status: 'draft'`) після кожного кроку. (Надійно, захищає від переривання зв'язку).

### ❓ 5.6. Механіка та тригери генерації стандартних запитань аватаром
*   **Питання:** Коли саме аватар генерує стандартні запитання на основі вмісту презентації та своєї ролі?
    *   *Варіант 1:* Автоматично один раз під час створення проекту/завантаження презентації.
    *   *Варіант 2:* Користувач натискає явну кнопку "Генерувати запитання" в інтерфейсі Train Mode.
    *   *Варіант 3:* Динамічно «на льоту» під час розмови (AI аналізує поточний відкритий слайд і створює запитання в реальному часі).
*   Чи використовуємо ми для аналізу контенту слайдів мультимодальні моделі (Vision API для аналізу зображень), чи аналізуємо суто текстові скрипти слайдів?

### ❓ 5.7. Голосове відтворення правильних відповідей у "Show Answer"
Якщо в Coach Settings для відображення правильної відповіді обрано параметр `показувати словами` (або голосом):
*   **Питання:** Чи повинна система динамічно генерувати TTS (Text-to-Speech) озвучку правильної відповіді голосом аватара або коуча, щоб учень міг її *прослухати*, чи це просто текстова підказка, яку продавець має сам прочитати вголос? (Генерація TTS потребує додаткових витрат лімітів ElevenLabs / Azure).

### ❓ 5.8. Валідація перемикання слайдів (Slide Navigation Constraint)
*   **Питання:** Що відбувається, якщо продавець відповів абсолютно правильно словами (аудіо/текст), але забув перемкнути презентацію на потрібний слайд (наприклад, `Prices`)?
    *   *Варіант А:* Зарахувати відповідь, але суттєво знизити загальний бал за невміння візуально підкріплювати слова слайдами.
    *   *Варіант В:* Зарахувати як повністю неправильну дію (0 балів), оскільки демонстрація слайда є обов'язковим критерієм.
    *   *Варіант С:* Автоматично перемкнути слайд на правильний за продавця та вивести попередження "Зверніть увагу на слайд".

### ❓ 5.9. Підтримка декількох правильних відповідей / слайдів на одне запитання
У реальних переговорах на одне запитання покупця (наприклад, *"Які умови оплати?"*) може бути декілька правильних варіантів відповідей або декілька релевантних слайдів (наприклад, слайд з планами підписок або слайд з реквізитами).
*   **Питання:** Чи підтримує наша архітектура зв'язок один-до-багатьох для правильних відповідей та очікуваних слайдів? Якщо так, таблицю `buyer_scenarios` потрібно буде доопрацювати, винісши очікувані слайди та відповіді в окрему зв'язану таблицю `buyer_expected_actions`.

### ❓ 5.10. Версійність RAG та інструкцій при перенавчанні
*   **Питання:** Що відбувається, коли експерт у Train Mode повторно відповідає на те саме запитання, але по-іншому?
    *   Попередня версія відповіді в RAG/інструкціях перезаписується повністю.
    *   Зберігається історія версій відповідей (з датами), щоб можна було повернутися до старих скриптів продажів.
    *   Створюється дублікат відповіді (система сприймає обидва варіанти як правильні).
```,StartLine:146,TargetContent:
