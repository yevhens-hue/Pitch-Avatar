# Epic: Universal AI Widget Integration Platform

## Goal
Розширити існуючий AI Widget, щоб він працював як незалежний продукт, який може інтегруватися з будь-якою веб-аплікацією через універсальний REST API. 

Widget не міститиме бізнес-логіку конкретного застосунку. Замість цього будується **двостороння інтеграція**: зовнішній застосунок передає Widget лише свій контекст (наприклад, поточну сторінку, події на кшталт `FIRST PROJECT = TRUE`) та конфігурацію, а Widget на основі своїх внутрішніх Trigger-ів та Instructions вирішує, які підказки дати, і надсилає назад до застосунку команди на виконання (відкрити модалку, запустити гід Stonly тощо).

---

## Business Value
- **Один AI Widget** для будь-якої веб-аплікації (CMS, Softprom, WordPress, Drupal).
- **Швидка реалізація (Перевикористання рушія):** Більшість епіка будується на вже наявному рушії (тригери, ланцюжки дій, доставка через WS), що суттєво зменшує ризики та час розробки.
- **No-code бізнес-логіка:** Можливість створювати нові сценарії (через інструкції, карти сайту, тригери) без зміни коду Widget та без дублювання складної бізнес-логіки у зовнішньому продукті.
- **Універсальність:** Віджет сам «прораховує» сценарії; зовнішня система лише надсилає свій стан (INBOUND) і виконує універсальні команди (OUTBOUND).
- **Зниження витрат:** Пишемо код один раз. Усувається потреба створювати унікальні віджети під різних клієнтів.
- **Продуктивність:** Оптимізація бандлу та швидкодії (очищення віджета від зайвого коду).

---

## Current Problem
Зараз Widget жорстко прив'язаний до логіки нашої CMS. Відсутній універсальний механізм INBOUND інтеграції, який дозволяє зовнішньому застосунку динамічно керувати віджетом: передавати поточний стан, контекст, чи запускати тригери.

**Наслідок:** 
- Для реалізації нових сценаріїв доводиться програмувати обробники логіки безпосередньо у продуктах-хостах.
- Якщо ми хочемо підключити сторонній сервіс (наприклад, Stonly), нам доводиться жорстко кодувати цей обробник у зовнішньому застосунку.
- Віджет містить багато коду або прихованих модулів (наприклад, плеєр слайдів), які не потрібні в режимі універсального асистента.

---

## Solution & Architecture Boundary
Побудувати універсальну двосторонню інтеграцію між Widget та зовнішнім застосунком. Важливо розуміти **межу відповідальності**: зовнішня система *повинна* мати API для надсилання подій та прийому команд, але вона *не повинна* містити саму логіку сценаріїв (if/else).

**1. Application → Widget (Inbound)**
Зовнішній застосунок через універсальний REST API передає:
- Зміну свого стану (наприклад, `CURRENT PAGE = PROJECT EDIT_SLIDES`, `FIRST PROJECT = TRUE`).
- Свою очікувану структуру даних та webhook URL (щоб віджет міг адаптуватися).
*(Widget приймає ці дані та використовує їх як умови для запуску своїх Trigger-ів).*

**2. Widget → Application (Outbound Connector)**
Widget використовує існуючий Outgoing Webhook (або новий адаптивний конектор), щоб смикати API зовнішнього продукту. Trigger-и визначають, яку послідовність дій виконати:
- Надіслати команду (наприклад, відкрити модалку або запустити Stonly-гід).
- Дочекатися завершення дії, якщо це необхідно для наступного кроку (MultiActions).

## Scope

### 1. INBOUND REST API
Реалізувати API для взаємодії Application → Widget:
- передача подій, зміна параметрів та контексту (які будуть слугувати тригерами для дій аватара).
- оновлення стану та конфігурації Widget (наприклад, з Enrollment порталу).

### 2. OUTBOUND Integration & Universal Connector
- Створити універсальний коннектор, який дозволяє екшенам віджета смикати зовнішні API (REST, Webhook, WebSocket).
- Віджет повинен мати можливість формувати Payload у тій структурі, яку очікує хост-система.

### 3. Backend (Go) та Переробка LLM
Більшість функціоналу базується на перевикористанні наявного бекенд-рушія (тригери, ланцюжки дій, WS). Нова робота на Go-стороні фокусується на:
- **Передачі повного документа** (опис URL сторінок, додаткова інформація) замість фрагментарних чанків контексту з RAG.
- **Гнучкій роботі з промптами та контекстом** (переробка системи взаємодії з LLM).
- Створенні нових типів Actions (конекторів) за потреби.

### 4. Trigger Engine & MultiActions
Тригери повинні вміти обробляти складні параметри.
*Приклад логіки всередині віджета:*
`Якщо (FIRST PROJECT == TRUE) та (NO ACTIONS == 20 sec) -> Дія: сказати "Створіть свій перший проект" -> Змінити параметр FIRST PROJECT = FALSE.`
Для навігації: карта сайту та екранів обробляється як повний документ, а не через стандартний RAG.

### 5. Widget Configuration
Гнучке налаштування інтерфейсу (можливо, через Enrollment портал):
- можливість приховувати непотрібні модулі (наприклад, текстовий чат).
- кастомізація UI для зовнішніх клієнтів (кольори, логотипи).

### 6. Widget Performance
- Фізично видалити з бандлу віджета модулі, що не використовуються (наприклад, показ слайдів), а не просто приховувати їх візуально.
- Скоротити час ініціалізації (враховуючи, що є невідворотна затримка від сторонніх сервісів ліпсинку, ми маємо оптимізувати все, що в наших силах).

## MVP User Stories

- **Navigation:** AI відповідає на питання користувача та відкриває необхідний екран застосунку (карта навігації закладена як повний документ). Віджет надсилає зовнішній системі стандартизовану команду переходу.
- **External Services (e.g. Stonly):** Замість хардкоду в CMS, віджет через тригер `open-guide` безпосередньо викликає необхідний API або передає команду зовнішньому додатку.
- **Knowledge:** AI відповідає на питання, використовуючи RAG.
- **Application Events:** AI реагує на події застосунку (хост через Inbound API повідомляє "Створено перший проєкт", віджет запускає вітальне відео).
- **Proactive Scenarios:** AI самостійно запускає сценарії (через таймери неактивності або зміну параметрів).

## Research Phase (Наступні кроки перед розробкою)
Оскільки це комплексна зміна, команді необхідно провести Research та фіналізувати технічне рішення:
1. **Архітектура INBOUND REST API** (як саме Application пушитиме дані у Widget).
2. **Механізм надійної двосторонньої взаємодії** (REST / Webhook / WebSocket) для передачі команд від Widget до Application (Connector).
3. **Формат передачі контексту в LLM** (повний документ карти сайту/навігації).
4. Оцінка можливості дочекатися результату виконання зовнішньої команди (Sync/Async MultiActions).

## Future Scope
Підтримати у майбутньому:
- LLM + Trigger Hybrid Flow;

- Public REST API, Public WebSocket API, SSE API;
- використання Widget як окремого SaaS-продукту (plug-and-play інтеграція з WordPress, Drupal, Shopify).

## Acceptance Criteria
- [ ] Проведено Research фазу та затверджено механізм двосторонньої взаємодії.
- [ ] Реалізовано універсальний INBOUND REST API для керування станом віджета ззовні.
- [ ] Реалізовано універсальний механізм (Connector/Webhook) для передачі команд від Widget до хоста.
- [ ] Backend: Перероблено систему роботи з LLM (передача повного документа замість чанків з RAG).
- [ ] Widget працює як незалежний продукт; бізнес-логіка сценаріїв винесена у Trigger-и та Instructions.
- [ ] Застосунок-хост може динамічно змінювати параметри Widget (вкл/викл модулів через Enrollment).
- [ ] Непотрібні модулі (плеєр слайдів) фізично видалені з бандлу віджета для оптимізації швидкодії.
- [ ] Поточний функціонал Widget залишається сумісним.
