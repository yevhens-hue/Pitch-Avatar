# Epic: Universal AI Widget Integration Platform (Hybrid Model)

## Goal
Розширити існуючий AI Widget, щоб він працював як незалежний продукт, який може інтегруватися з будь-якою веб-аплікацією через універсальний REST API. 

Перехід на **Гібридну модель (Тригери + LLM)**: побудова єдиного шару здатностей (Capability Layer), де критичні бізнес-процеси керуються жорсткими детермінованими правилами, а відкрита навігація та допомога — контрольованою автономією LLM.

## Business Value
- **Один AI Widget для будь-якої веб-аплікації** (CMS, Softprom, WordPress, Drupal).
- **Спільний Capability Layer**: Екшени та інструменти пишуться один раз і доступні як для жорстких тригерів, так і для LLM. Ми не дублюємо логіку.
- **Безпечна автономія**: Завдяки системі "огорож" (Guardrails) ми отримуємо гнучкість штучного інтелекту без ризику галюцинацій на критичних етапах (наприклад, видалення акаунту чи білінг).
- **Готовність до MCP (Model Context Protocol)**: Закладання відкритого стандарту вже зараз дозволить легко підключати зовнішні інструменти (PA-DeckJam) та розширювати екосистему без переписування плеєра.
- **Зниження витрат та продуктивність**: Оптимізація бандлу, видалення зайвого коду (слайд-плеєр), єдина точка підтримки інтеграцій.

## Current Problem
Зараз Widget жорстко прив'язаний до логіки CMS. Відсутній універсальний механізм INBOUND інтеграції для динамічного керування віджетом. 
Для нових сценаріїв (напр. підключення Stonly) доводиться хардкодити обробники у продуктах-хостах. Плеєр перевантажений модулями, які не потрібні для режиму універсального асистента.

## Solution & Architecture Boundary

Архітектура будується навколо концепції **Спільного Capability Layer** та **Гібридного запуску**.

### 1. Shared Capability Layer (Спільний шар здатностей)
Замість розрізнених API створюється єдиний шар інструментів (Actions/Tools) — наприклад: `navigate`, `open_guide`, `speak`, `create_project`. Цей шар з самого початку проектується як **MCP-сумісний**. 

### 2. Hybrid Launch Mechanism (Два способи запуску)
Над Capability Layer працюють два механізми вибору дій (різниця лише в тому, ХТО обирає дію, виконання — однакове):
*   **Детерміновані тригери (Правило → Action):** Запуск за параметром, голосом або подією. Гарантований, передбачуваний результат. Використовується для критичних дій (lead form, leave), одноразових подій (привітання) та комплаєнсу.
*   **LLM-Автономія (Контекст → Tools):** LLM самостійно обирає інструмент із Capability Layer на основі запиту користувача. Використовується для відкритої навігації ("як зробити X?"), багатокрокової допомоги та розмовного створення проєктів.

### 3. Safety Guardrails (Огорожі для LLM)
Щоб автономія була безпечною, запроваджуються жорсткі рамки:
*   **Bounded toolset:** LLM бачить лише дозволені здатності (allow-list на інтеграцію).
*   **Backend-валідація:** Кожну запропоновану дією LLM перевіряє бекенд (напр. `delete_account` — блок, `fill_details` — ок).
*   **Per-page контекст:** На кожній сторінці інжектиться контекст "що тут можна зробити" — LLM обирає лише з доступного.
*   **Детерміноване виконання:** Після вибору інструменту, він відпрацьовує як фіксований ланцюг MultiActions.

### 4. Взаємодія з Хостом
*   **Application → Widget (Inbound):** Передача подій (напр. FIRST PROJECT = TRUE), контексту та конфігурації.
*   **Widget → Application (Outbound Connector):** Widget смикає API зовнішнього продукту через стандартизований конектор.

## Scope

1. **Capability Layer & MCP-compatibility:** Створення шару інструментів. Адаптація плеєра до стандарту MCP (Model Context Protocol).
2. **INBOUND REST API:** Механізм передачі подій, зміни параметрів та per-page контексту від застосунку до Widget-а.
3. **OUTBOUND Universal Connector:** Конектор для виконання команд на стороні хост-системи. Віджет формує Payload у структурі, яку очікує хост.
4. **Hybrid Trigger Engine:** Оновлення бекенду (Go) для підтримки одночасної роботи детермінованих правил та LLM над єдиним пулом інструментів. Передача повного документа навігації замість фрагментарного RAG.
5. **Widget Configuration & Performance:** Фізичне видалення непотрібних модулів (слайди), кастомізація UI, мінімізація часу ініціалізації.

## MVP User Stories

*   **Navigation (LLM-Автономія):** AI відповідає на питання користувача та відкриває необхідний екран застосунку (карта навігації закладена як повний документ у контекст). Віджет обирає інструмент `navigate` з Capability Layer і надсилає зовнішній системі стандартизовану команду переходу.
*   **External Services (Capability Layer, e.g. Stonly):** Замість хардкоду в CMS, віджет через тригер/інструмент `open-guide` безпосередньо викликає необхідний API або передає команду зовнішньому додатку.
*   **Knowledge (RAG):** AI відповідає на питання, використовуючи базу знань (RAG).
*   **Application Events (Детерміновані тригери):** AI реагує на події застосунку (хост через Inbound API повідомляє "Створено перший проєкт", віджет через детермінований тригер запускає вітальне відео без участі LLM).
*   **Proactive Scenarios (Детерміновано / Гібрид):** AI самостійно запускає сценарії (через таймери неактивності або зміну параметрів).

## Research Phase
- Детальний мапінг існуючих екшенів у формат MCP Tools.
- Архітектура INBOUND REST API (формат пушу даних у Widget).
- Дизайн системи Backend-валідації для Bounded toolset.

## Future Scope
- "Асистент, що налаштовує асистента": користувач описує бажану поведінку текстом, а ШІ сам збирає детерміновані тригери.
- Розмовне створення проєктів (збір параметрів через діалог).
- Public REST API, Public WebSocket API, SSE API.
- Plug-and-play інтеграція з WordPress, Drupal, Shopify.

## Acceptance Criteria
- [ ] Спроектовано та впроваджено спільний Capability Layer, закладено архітектуру MCP-сумісності у плеєрі.
- [ ] Реалізовано Гібридний рушій: критичні події обробляються детермінованими тригерами, а відкрита навігація — LLM-автономією з використанням Bounded tools.
- [ ] Впроваджено систему безпеки (Guardrails): Backend-валідація дій LLM та Per-page контекст.
- [ ] Реалізовано універсальний INBOUND REST API для керування станом віджета ззовні.
- [ ] Реалізовано універсальний OUTBOUND Connector для передачі команд від Widget до хоста.
- [ ] Backend: Перероблено систему роботи з LLM під використання Tools замість простої генерації тексту.
- [ ] Непотрібні модулі (плеєр слайдів) фізично видалені з бандлу віджета для оптимізації швидкодії.
