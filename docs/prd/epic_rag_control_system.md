## 1. Overview

### 1.1. Goal & Description
Дать презентаторам и владельцам проектов гранулярный контроль над тем, как AI-аватар формирует ответы: разрешить жесткую привязку исключительно к базе знаний (RAG), разрешить гибридный режим (RAG + LLM) или полностью отключить базу знаний (LLM Only). Дополнительно — реализовать возможность бесшовного подключения **внешних баз знаний клиента (External RAG)** для Enterprise-сценариев.

### 1.2. Business Value
*   **Снижение галлюцинаций до 0%:** В режиме *Grounded* аватар гарантированно не будет выдумывать ответы для специализированных проектов (например, техподдержка или Legal), что критично для B2B-сегмента.
*   **Привлечение Enterprise-клиентов:** Возможность прямой интеграции (External RAG) позволяет корпорациям использовать свои существующие, секьюрные базы знаний без необходимости дублировать документы на серверах Pitch Avatar.
*   **Повышение Retention:** Гибкость настройки под разные Use Cases (от строгого корпоративного ассистента до креативного собеседника) увеличивает ценность продукта для широкого круга пользователей.

## 3. User Story & Problems

### 3.1. Problems
1. **Галлюцинации вне контекста:** Использование общих знаний LLM недопустимо для узкоспециализированных ботов.
2. **Своя инфраструктура:** У крупных клиентов уже есть свои RAG-решения, которые они хотят использовать как источник истины.

### 3.2. User Story
> **Как** презентатор (владелец проекта),
> **Я хочу** иметь возможность настраивать режим ответов аватара (только по RAG, RAG + LLM, только LLM) и подключать внешний RAG по API,
> **Чтобы** полностью контролировать релевантность ответов и использовать существующую инфраструктуру знаний моей компании для различных Use Cases.

## 4. Implementation

В разделе настроек **Knowledge Base** (База знаний) необходимо добавить два новых функциональных блока: **Answer Mode** и **External RAG Connection**.

### 4.1. Режимы ответов (Answer Mode)
Пользователь может выбрать один из трех режимов работы:

1. **Grounded (Строгий RAG):** 
   * Ответ формируется *исключительно* на основе найденных источников (внутренних или внешних).
   * Если релевантных источников нет — аватар честно отвечает заготовленной фразой (например, "Извините, у меня нет информации об этом"). Знания LLM блокируются.
2. **Hybrid (Смешанный):** 
   * Система сначала ищет данные в RAG. 
   * LLM формирует ответ на базе найденных фактов, но имеет право дополнить ответ своими общими знаниями, если запрос этого требует.
3. **LLM Only (Без базы знаний):** 
   * Поиск по RAG (внутреннему или внешнему) не производится вообще. Аватар отвечает, опираясь исключительно на свой промпт и базовые знания модели.

### 4.2. Подключение внешней базы (External RAG Connection)
Интерфейс для настройки подключения к стороннему RAG-провайдеру клиента. 

**Настраиваемые поля (Форма):**
*   **Connection Name:** Пользовательское имя (напр. *“Client Support RAG”*)
*   **API Endpoint:** URL-адрес клиентского endpoint'а.
*   **Authentication (Тип авторизации):**
    *   API Key / Bearer Token / OAuth / No Auth
*   **Timeout:** Настройка таймаута ожидания ответа (например, 3–10 секунд).

**Request Format (Что мы отправляем клиенту):**
*   `query` (Текст запроса пользователя)
*   `user_language` (Язык запроса)
*   `conversation_history` (История диалога для контекста)
*   `avatar_id` (Идентификатор аватара)
*   `customer/session_id` (Идентификатор сессии/пользователя)

**Response Format (Что мы ожидаем от клиента):**
*   `answer` (Синтезированный ответ)
*   `chunks` (Найденные куски текста)
*   `sources` (Ссылки на источники)
*   `confidence_score` (Уверенность внешнего RAG в ответе)
*   `metadata` (Дополнительные данные)

### 4.3. Настройка Fallback-поведения (Safety)
Система должна уметь корректно обрабатывать нестандартные ситуации:
*   Если внешний RAG **не отвечает** (по таймауту).
*   Если возвращенный `confidence_score` **ниже заданного порога** (threshold).
*   Если релевантные **источники не найдены**.
*(Поведение в этих случаях зависит от выбранного Answer Mode: в Grounded-режиме аватар извинится, в Hybrid — может попробовать ответить сам).*

## 5. Workflow (Пример логики работы)

1. **Входной запрос:** Пользователь (зритель) задает вопрос аватару.
2. **Анализ настроек:** Сервер Pitch Avatar определяет выбранный `Answer Mode` для этого проекта.
3. **Поиск (если нужно):** 
   * Если режим *не* LLM Only, система проверяет наличие External RAG.
   * Если External RAG включен ➔ Отправляет `query` и контекст по указанному API Endpoint'у.
4. **Получение данных:** Pitch Avatar получает в ответ `chunks`, `answer`, `sources` и `confidence_score`.
5. **Генерация финального ответа:** LLM Pitch Avatar формирует итоговый текст ответа:
   * **[Grounded]** ➔ Выдает ответ *только* если есть `sources`. Иначе — fallback фраза.
   * **[Hybrid]** ➔ Интегрирует `chunks` с общими знаниями LLM.
   * **[LLM Only]** ➔ (Шаги 3-4 пропускаются), отвечает как обычный ChatGPT.
6. **Вывод:** Аватар произносит/печатает ответ.

## 7. Implementation Phases

### Phase 1: Foundation & Base UI (Спринт 1)
**Фокус:** Настройка базы данных и интерфейса управления режимами.
*   Обновление схемы БД (настроек проекта) для сохранения полей `answer_mode` и конфигурации External RAG.
*   Верстка карточек Answer Mode (Grounded, Hybrid, LLM Only) в интерфейсе Knowledge Base согласно прототипу.
*   Верстка формы подключения External RAG (Endpoint, Auth, Timeout).

### Phase 2: RAG Modes Implementation (Спринт 2)
**Фокус:** Адаптация логики внутреннего поиска под новые режимы.
*   Модификация бекенд-пайплайна генерации ответов.
*   Реализация строгого режима **Grounded** (настройка промптов LLM для игнорирования общих знаний и выдачи fallback-фразы при отсутствии релевантных чанков).
*   Реализация логики **LLM Only** (обход этапа поиска по базе знаний).

### Phase 3: External RAG Pipeline (Спринт 3)
**Фокус:** Сетевое взаимодействие со сторонними базами.
*   Реализация отправки HTTP-запросов (Payload) на указанный клиентом Endpoint с нужным типом авторизации.
*   Обработка таймаутов и неудачных ответов (Graceful degradation).
*   Парсинг ответа от внешнего RAG (маппинг `chunks`, `confidence_score`) и передача их в нашу LLM для финального синтеза.

### Phase 4: QA, Polish & Analytics (Спринт 4)
**Фокус:** Безопасность, метрики и релиз.
*   Интеграция событий в PostHog (настройки сохранены, внешний запрос успешен/упал, сработал fallback).
*   Тестирование на мок-серверах External RAG.
*   Внедрение ползунка Confidence Threshold для тонкой настройки гибридного режима.
