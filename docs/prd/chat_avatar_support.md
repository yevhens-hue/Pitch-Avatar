## 1. Overview

### 1.1. Description
Разработка и внедрение глобального плавающего виджета ИИ-ассистента "Sara", который заменит (или дополнит) классические статические чеклисты и видео-уроки. Ассистент учитывает текущий экран пользователя и его бизнес-цель (`main_goal`), генерируя релевантные подсказки и быстрые вопросы. Бот обладает возможностью напрямую запускать интерактивные туры Userflow из диалога, превращая пассивную справку в активного помощника.

### 1.2. Business Value
*   Снижение drop-off rate на ключевых экранах флоу создания контента на 15-20%.
*   Повышение вовлеченности (Engagement rate) за счет персонализации на основе `main_goal`.
*   Улучшение конверсии в активацию (`user_activated`) за счет бесшовной связи ИИ-консультаций с интерактивными турами.

## 3. User Stories (US)

### UI Components & Interaction States (Структура «Формы»)
На основе дизайна ассистента "Sara", виджет состоит из следующих функциональных блоков:

#### 1. Header
*   **Context Title:** Динамический текст, привязанный к текущему разделу (например, `Sara · Step 1: Upload Slides`).
*   **Sound Toggle:** Кнопка управления голосовым сопровождением (🔊/🔇).
*   **More Actions (⋯):** Кнопка вызова меню-переполнения. Сюда перенесен функционал управления вложениями (Attachment).
*   **Close Control (X):** Единственная кнопка управления состоянием в шапке.
    *   **Full Dismiss:** Нажатие на X полностью закрывает виджет и скрывает плавающую кнопку (FAB).
    *   **Minimize:** Клик вне области диалогового окна автоматически сворачивает его в FAB. Кнопка «минус» (свернуть) удалена для упрощения интерфейса.

#### 2. Avatar Area
*   **Status Indicator:** Текст состояния (например, `Sara · Ready to help`, `Speaking...`, `Thinking...`) с анимированным индикатором.
*   **Character:** Центральное изображение/анимация ассистента, меняющееся в зависимости от состояния (idle/thinking/speaking).

#### 3. Chat Interface
*   **Messages:** Диалоговые баблы. Сообщения бота — слева (с аватаром-буквой "S"), пользователя — справа (акцентный цвет).
*   **Suggested Chips (с пагинацией):** Динамические кнопки быстрых ответов.
    *   Отображаются по 3 штуки за раз.
    *   Добавлена кнопка ↺ (Refresh) для циклической смены набора чипов (всего до 6 подсказок на один контекст).

#### 4. Input Panel
*   **Voice (Mic):** Кнопка активации микрофона для голосового ввода, расположена слева.
*   **Text Input:** Поле ввода с плейсхолдером `Ask Sara anything...`.
*   **Send:** Кнопка отправки сообщения, расположена справа.

### US-01: Глобальный контейнер виджета и Инициализация
**User Story:** Как система, я хочу рендерить плавающий виджет поверх всех окон приложения, чтобы пользователь всегда имел доступ к помощнику.

**Acceptance Criteria:**
*   Компонент `SaraWidget` интегрируется в корень приложения.
*   Виджет загружается лениво (`next/dynamic` с `ssr: false`), чтобы не блокировать LCP и TBT метрики.
*   Рендеринг происходит только при активации фича-флага `posthog.isFeatureEnabled('chat-avatar-support')`.
*   При первой загрузке компонента отправляется событие `chat_avatar_rendered`.
*   При клике на иконку отправляется событие `chat_avatar_opened` с параметрами: `{ user_id, main_goal, screen: window.location.pathname }`.
*   Состояние чата (открыт/закрыт, история сообщений) сохраняется в глобальном хранилище (например, Zustand `useSupportChatStore`).

### US-02: Окно диалога и Контекстность (Expanded Chat Window)
**User Story:** Как пользователь, я хочу видеть подсказки и быстрые вопросы, которые релевантны тому экрану, где я сейчас нахожусь, чтобы не тратить время на ввод текста.

**Acceptance Criteria:**
*   В шапке развернутого виджета отображается контекстный подзаголовок, зависящий от текущего URL (например, «Step 1: Upload Slides»).
*   Предложенные вопросы (Suggested chips) динамически генерируются с учетом текущего экрана и `main_goal` пользователя из профиля.
*   Панель ввода поддерживает текст и голосовую запись. Функция загрузки вложений вынесена в меню настроек (⋯) в шапке виджета.
*   При отправке сообщения отправляется событие `chat_avatar_message_sent`.
*   Пока ожидается ответ от бекенда, отображается индикатор `typing...` и анимация (пульс) на аватаре.

### US-03: Интеграция с Userflow (Launcher Module)
**User Story:** Как пользователь, я хочу, чтобы бот мог автоматически показать мне, куда нажимать в интерфейсе, чтобы мне не пришлось искать нужные кнопки самому.

**Acceptance Criteria:**
*   Бот может возвращать HTML-разметку с кнопками-CTA, содержащими атрибуты `data-tour-id` и `data-screen`.
*   JS-обработчик внутри виджета перехватывает клики по этим кнопкам.
*   **Роутинг:** Если `data-screen` не равен текущему URL, вызывается `navigateTo()`. После задержки в 800ms для рендера, вызывается `Userflow.startTour(tourId)`.
*   Если редирект не требуется, тур запускается сразу (без 800ms задержки).
*   **Graceful fallback:** Если объект `window.Userflow` не определен, осуществляется только роутинг на нужный экран без падения UI.
*   В PostHog отправляется событие `chat_avatar_tour_triggered` с параметром `{ tour_id }`.
*   **Mobile:** При запуске тура на мобильных устройствах, окно чата должно автоматически сворачиваться, чтобы не перекрывать подсвеченные Userflow элементы интерфейса.
*   Предусмотрен сценарий Fallback (Handoff): при запросе 'human support' или 'operator' бот инициирует открытие стандартного виджета Stonly (`window.openStonlyGuide`).

### US-04: Проактивные триггеры (Proactive Messaging Logic)
**User Story:** Как пользователь, я хочу, чтобы Сара сама предлагала помощь, когда я сталкиваюсь с трудностями или успешно завершаю этап, чтобы я чувствовал поддержку на каждом шагу.

**Acceptance Criteria:**
*   **Time-based:** Если пользователь находится на сложной форме (Editor/Wizards) без активности более X секунд, Сара показывает баббл с вопросом: «Нужна помощь с настройкой?».
*   **Error-based:** При возникновении ошибок валидации или сбоев API, Сара проактивно предлагает решение или ссылку на FAQ.
*   **Success-based:** После успешного выполнения ключевого действия (например, экспорт видео), Сара хвалит пользователя и предлагает следующий логический шаг.
*   **Visual Cue:** Проактивное сообщение отображается в виде всплывающего "баббла" над иконкой FAB, не открывая чат полностью до клика пользователя.
*   **Anti-Spam:** Внедрена логика Cooldown, чтобы бот не писал пользователю чаще чем раз в N минут.

## 4. Implementation Phases

### Phase 1: Foundation & Basic UI (Спринт 1)
**Фокус:** Создание базового компонента, стейт-менеджмента и аналитики.
*   Настройка Zustand хранилища (`useSupportChatStore`) для сохранения состояния между страницами.
*   Верстка компонента `SaraWidget` (свернутое и развернутое состояние, анимации).
*   Интеграция с PostHog: проверка фича-флага `chat-avatar-support`.
*   Ленивая загрузка виджета в `layout.tsx`.
*   Настройка базовой аналитики (`chat_avatar_rendered`, `chat_avatar_opened`).

### Phase 2: Context & RAG Integration (Спринт 2)
**Фокус:** Контекстность, предложенные вопросы и работа с бекендом.
*   Написание логики определения контекста (чтение `pathname` из Next.js и `main_goal` из `AuthContext`).
*   Динамическая генерация Suggested Chips в зависимости от текущего экрана.
*   Реализация панели ввода (текст) и индикатора `typing...`.
*   Подключение к API бекенда (отправка контекста и получение текстового ответа от ИИ).
*   **Разработка механизма отслеживания бездействия пользователя (Time-based trigger) для вызова проактивного баббла.**

### Phase 3: Userflow Tour Launcher (Спринт 3)
**Фокус:** Бесшовный запуск туров из интерфейса чата.
*   Парсинг ответов ИИ: перехват кнопок-CTA с атрибутами `data-tour-id` и `data-screen`.
*   Реализация логики навигации (`router.push`) с задержкой 800ms для рендера.
*   Интеграция с объектом `window.Userflow` для запуска тура.
*   Доработка мобильного поведения: принудительное сворачивание чата при старте тура, чтобы не перекрывать подсказки Userflow.
*   **Реализация триггеров на основе событий (Error-based и Success-based) и настройка логики Cooldown (анти-спам).**

### Phase 4: QA, Polish & Release (Спринт 4)
**Фокус:** Стабилизация и раскатка.
*   Настройка голосового ввода (Voice-to-Text) и поддержки вложений (если требуется в v1.0).
*   Финальное тестирование на Staging (проверка fallback-поведения при отсутствии Userflow).
*   Проверка полноты данных в PostHog (event properties).
*   Включение фича-флага на 10-20% аудитории для замера метрик.
