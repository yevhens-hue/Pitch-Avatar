# Эпик: Архитектура управления биллингом и тарифными планами

## Описание

### 1. Обзор
**Название Эпика:** Реализация управления биллингом и тарифных планов
**Product Owner:** yevhen.shaforostov
**Основано на:** [PRD v.1.0] Управление биллингом и тарифные планы
**Статус:** 🟢 В работе

#### 1.1. Описание
Разработка нового раздела управления биллингом и рефакторинг уже существующей страницы тарифов (базовый функционал которой уже реализован на `https://app.pitchavatar.com/main/plans`).

**Задача включает:**
*   **Создание единого Billing Dashboard:** Проектирование прозрачного центра управления финансами, где критические действия (обновление карты) вынесены на первый экран для предотвращения неуспешных платежей (Failed Payments).
*   **Рефакторинг Pricing Plans:** Интеграция существующей матрицы тарифов в новый интерфейс настроек с внедрением модального окна для сбора Enterprise-лидов.
*   **Архитектурную изоляцию данных:** Вынос всей логики (пагинация инвойсов, расчет лимитов для прогресс-баров) в хук `useBillingData` для полной готовности к подключению реального API.
*   **Глобальную систему уведомлений:** Внедрение механизмов (Zustand) для перехвата статуса `past_due` и отображения красных баннеров об ошибке оплаты поверх всего интерфейса приложения.
*   **Бухгалтерскую прозрачность:** Обеспечение возможности просматривать полный архив инвойсов и экспортировать историю платежей в формате CSV.

#### 1.2. Бизнес-ценность
*   **Self-Service Billing:** Единая панель для истории платежей и инвойсов без обращений в поддержку.
*   **Визуализация лимитов:** Процесс исчерпания лимитов становится очевидным и предсказуемым благодаря прогресс-барам.
*   **Улучшение конверсии (Upgrade rate):** Прозрачная страница сравнения тарифов с удобным переключением периодов оплаты.
*   **Генерация Enterprise-лидов:** Удобная форма сбора контактов прямо внутри тарифной сетки.
*   **Снижение оттока (Churn Prevention):** Прозрачная система управления картой и возможность быстрой отмены подписки без негатива.
*   **Стимулирование доп. продаж (Add-ons):** Прямые ссылки на покупку дополнительных пакетов ресурсов.
*   **Ускорение Time-to-Market:** Четкое разделение UI и бизнес-логики через `useBillingData`.

#### 1.3. Структура продукта и флоу
*   **Схема структуры:** Настройки биллинга в `Settings` -> вкладка `Billing`. Страница планов -> `Plans`.
*   **Прототип:** [Pitch Avatar Lab Settings](https://pitch-avatar-lab.vercel.app/settings)

### 3. User Stories (US)

#### Компоненты UI и состояния взаимодействия
На основе бизнес-требований, интерфейс состоит из следующих блоков:

**1. Hero Section (Панель верхнего уровня)**
*   **Дата и сумма следующего платежа:** Крупный, предсказуемый индикатор предстоящего списания.
*   **Маска активной карты:** Визуализация привязанной карты (например, `Visa **** 4242`).
*   **Действия:** Кнопки "Update Card" и "Billing Details", ведущие напрямую в портал PayPro.

**2. Прогресс-бары использования**
*   **Компонент Bar:** Окрашивается в предупреждающие цвета при достижении 80% лимита.
*   **Ссылки на аддоны:** Ссылки на покупку минут прямо рядом с соответствующим прогресс-баром.

**3. Таблица истории**
*   **Список:** Список всех инвойсов с деталями.
*   **Пагинация:** Кнопка "Load More".
*   **Экспорт:** Кнопка скачивания всего реестра платежей в формате CSV.

**4. Система глобальных уведомлений**
*   **Баннер:** Красный Alert в шапке приложения (`layout.tsx`), оповещающий о неудачной транзакции или скором списании.

---

#### US-01: Hero Section и быстрое управление картой
**User Story:** Как владелец аккаунта, я хочу видеть на первом экране, какая карта привязана и когда будет следующее списание, чтобы быть уверенным в наличии средств и при необходимости изменить карту в 1 клик.

**Acceptance Criteria:**
*   Отображается `activeCardMask` (если данные доступны из API).
*   Ссылки "Update Card" используют переменную `NEXT_PUBLIC_PAYPRO_CHANGE_CARD_URL`.
*   При клике отправляется событие аналитики `update_card_clicked`.

#### US-02: Предсказуемые инвойсы и экспорт
**User Story:** Как бухгалтер B2B-клиента, я хочу скачать всю историю платежей в одну таблицу Excel, чтобы не скачивать каждый PDF по отдельности.

**Acceptance Criteria:**
*   Таблица рендерит элементы типа `PaymentHistoryItem`.
*   Кнопка "Export to CSV" генерирует файл на стороне клиента и инициирует загрузку.
*   Клик по иконке PDF инициирует загрузку одиночного инвойса.
*   Отправляется событие `history_csv_exported`.

#### US-03: Глобальные уведомления (Failed Payments)
**User Story:** Как пользователь, чья карта просрочена или заблокирована, я хочу сразу увидеть сообщение об ошибке на дашборде, чтобы быстро обновить данные и не потерять доступ к сервису.

**Acceptance Criteria:**
*   Если `paymentStatus === 'past_due'`, рендерится глобальный красный баннер.
*   Баннер содержит прямую ссылку на страницу обновления карты.
*   Состояние считывается из глобального стора Zustand (`useAppStore`).

### 4. Фазы реализации

#### Фаза 1: Фундамент и слой данных (Спринт 1)
**Фокус:** Создание хуков, стейт-менеджмента и базовой верстки.
*   Разработка `useBillingData.ts` и описание типов данных.
*   Настройка ENV-переменных для интеграции с PayPro.
*   Создание глобального стора (Zustand) для хранения `paymentStatus`.
*   Верстка Skeleton Loaders для имитации загрузки данных.

#### Фаза 2: Рефакторинг основного UI (Спринт 2)
**Фокус:** Визуализация лимитов и тарифов.
*   Внедрение компонента Usage Progress Bars (с логикой предупреждающих цветов).
*   Обновление сетки тарифов `PricingTable` (переключение Monthly/Annually).
*   Интеграция модального окна Enterprise Request Modal.

#### Фаза 3: Продвинутые функции для экспертов (Спринт 3)
**Фокус:** Бухгалтерская прозрачность и управление картами.
*   Внедрение компонента `CardMask` в Hero-блок.
*   Реализация утилиты `exportHistoryToCSV(historyData)`.
*   Разработка пагинации (кнопка Load More) для полного архива инвойсов.

#### Фаза 4: Проактивные алерты и QA (Спринт 4)
**Фокус:** Глобальные баннеры и финальный релиз.
*   Разработка `GlobalPaymentBanner` для отображения статусов `failed` / `past_due` в корневом `layout.tsx`.
*   Тестирование сценария Pre-charge Alerts (уведомления за 3 дня до списания).
*   Внедрение аналитики PostHog на все ключевые кнопки действий.
