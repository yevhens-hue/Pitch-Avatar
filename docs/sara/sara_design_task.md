### О проекте (Context & Goals)

Мы внедряем глобальный плавающий виджет ИИ-ассистента «Sara», который заменяет статические чеклисты и видеоуроки. Согласно **Epic R4C-25761**, виджет должен работать как **независимый универсальный продукт**, который интегрируется с любым веб-приложением (Host App) через REST API. 

Ассистент генерирует релевантные подсказки на основе контекста и системных событий, **переданных ему извне (Host App) через REST API**. Виджет не содержит бизнес-логики конкретного приложения (Pitch Avatar) и полностью настраивается через конфигурацию и триггеры.

**Техническое ограничение первого этапа (Важно!):** Аватар не может сам нажимать кнопки за пользователя. Его главный инструмент — это текст и **текстовые гиперссылки / кнопки действий** внутри диалога, клик по которым не меняет состояние внутри виджета, а отправляет команду (Webhook) в Host App, которое само открывает нужные модальные окна или разделы. В диалоге **не будет** кнопок-чипсов (Suggested Chips) для ввода.

**Бизнес-цель:** Снижение drop-off rate на 15-20%, улучшение конверсии в активацию за счет контекстной помощи.

### Где это находится

Виджет является независимым встраиваемым модулем и рендерится поверх всех окон любого веб-приложения (Host App), куда он интегрирован (обычно в правом нижнем углу).

### Ссылки и референсы

* **Визуальная основа (Бейзлайн — главное!):** За основу берётся уже готовый AI-ассистент, работающий на главной странице [pitchavatar.com](https://pitchavatar.com/). Его UI **является эталоном** — layout, стиль бабблов, расположение элементов, кнопки ввода. **От него нужно отталкиваться максимально близко**, адаптируя только под задачи внутреннего инструмента.
* **Рабочий прототип:** https://pitch-avatar-lab.vercel.app/
* **Референс по UX:** Intercom Messenger (Best Practices for Chat UI).
* **Спецификация:** Epic R4C-25761: AI Widget Assistant with Application Context & UI Navigation (Universal Widget).

### Что нужно нарисовать (Scope of Work)

#### Блок 1: Верхняя панель (Header)
**Задача:** Брендинг + закрытие виджета. Брендинг должен быть настраиваемым.
**Элементы (согласно эталону):**
* **Логотип** слева: иконка + текст бренда (передается через API Host-приложением). По умолчанию может быть логотип клиента.
* **Close (×):** Кнопка закрытия виджета — правый верхний угол. Это единственный контрол управления в хедере.
* *Кнопку «свернуть» (−) и кнопку «⋯» (More Actions) — **в хедер не включать**.*

#### Блок 2: Область аватара (Avatar Area)
**Задача:** Отображение видео-аватара + управление звуком.
**Элементы (согласно эталону):**
* **Video/Character:** Видео аватара на нейтральном/слегка размытом фоне. Занимает ~50% высоты виджета. Контент видео загружается динамически в зависимости от конфига.
* **Sound Toggle:** Иконка 🔇 / 🔊 в **верхнем правом углу** зоны аватара (поверх видео). Нужно показать оба состояния: вкл/выкл.
* **Avatar Name Badge:** Тёмный pill-тег с именем (например, `Sara` или кастомное имя) в **нижнем левом углу** зоны видео.
* **Состояния аватара:** Нужно продумать визуальное отличие состояний `idle` / `thinking` / `speaking` (может быть через border, badge-статус или микроанимацию).

#### Блок 3: Интерфейс чата (Chat Interface)
**Задача:** Отображение переписки.
**Элементы (согласно эталону):**
* **Бабблы бота (левое выравнивание):**
  * Стиль: **outline** — светло-голубая/синяя рамка, белый/прозрачный фон внутри.
  * Структура: имя бота + timestamp сверху, затем текст сообщения.
* **Бабблы пользователя (правое выравнивание):**
  * Стиль: белый/светло-серый фон, скруглённые углы. **Без акцентного цвета**.
  * Структура: метка (`Вы`) + timestamp сверху, текст ниже.
* **Индикатор печатания:** Анимация `...` (три точки) внутри баббла бота во время ожидания ответа.
* **Текстовые ссылки и Кнопки действий (Inline Action Buttons) — особый акцент:**
  * Интерактивные элементы встраиваются прямо в текст баббла бота.
  * **Обычные ссылки:** Стилизовать как кликабельные: акцентный цвет + подчёркивание + **иконка внешней ссылки** (квадратик со стрелочкой) в конце. Пример: *«[https://example.com ↗]»*.
  * **Action-кнопки (Навигация/Действия):** Отрисовать в виде кнопок (например, outline button синего цвета со скругленными углами) прямо внутри баббла или сразу под текстом. При клике на такую кнопку виджет **отправляет OUTBOUND Webhook** в Host App, а не выполняет переход сам. Дизайн должен призывать к целевому действию.

#### Блок 4: Панель ввода (Input Panel)
**Задача:** Работа с текстом и голосом.
**Элементы (согласно эталону):**
* **Поле ввода текста:** Без видимой рамки / border. Только placeholder-текст `Send a message` (серый). При фокусе можно добавить лёгкую обводку.
* **Кнопка отправки:** **Круглая, акцентного цвета (кастомизируемого)**, иконка стрелки вверх (↑). Располагается по центру-справа области ввода.
* **Кнопка микрофона:** Outlined иконка 🎙 справа от кнопки отправки.
  * *Голосовой режим (Continuous Voice):* При активации — пульсация или визуализатор волн, показывающий, что система слушает (Barge-In).

#### Блок 5: Проактивные триггеры и свернутое состояние (Proactive Bubbles)
**Задача:** Привлечение внимания пользователя до открытия чата.
**Элементы:**
* **FAB (Плавающая кнопка):** Свёрнутое состояние виджета — иконка (настраиваемая) в правом нижнем углу экрана. Отрисовать возможность полного скрытия (`hideFab: true`).
* **Баббл-уведомление:** Всплывающее облачко над FAB с текстом, инициированным системой.
  * *Внешний триггер:* Баббл может появляться по команде от Host App (INBOUND REST API), например, при долгом бездействии (событие `idle`) или системной ошибке.

#### Блок 6: Системные уведомления и модалки (Overlays)
**Задача:** Реакция виджета на действия пользователя, требующие подтверждения.
**Элементы (согласно эталону):**
* **Модальное окно (Modal Dialog):**
  * Всплывает по центру виджета, затемняя весь фон позади (overlay).
  * Пример: Предупреждение о переходе на внешний сайт.
  * Элементы: Кнопка закрытия (×) в правом верхнем углу, заголовок, текст с гиперссылкой, и две кнопки действий (второстепенная — outline «Оставайся здесь», главная — тёмная/залитая «Перейти на сайт»).

#### Блок 7: Адаптив (Mobile Fallback)
* При выполнении действий, перекрывающих экран в Host App (например, старт интерактивного тура), развёрнутое окно чата должно сворачиваться в FAB. Это будет вызываться через INBOUND REST API командой от Host App.

#### Блок 8: Context Awareness (Context-aware UI)
**Задача:** Визуально (или логически) отразить то, что ассистент понимает контекст текущей страницы, основываясь на данных, **переданных от Host App через API**.
**Элементы (на усмотрение дизайнера):**
* **Dynamic Welcome Text:** Приветственное сообщение при пустом чате должно меняться в зависимости от контекста (`config.contextLabel`), переданного по API. 
* **Context Badge (Опционально):** Небольшой бейдж или статус-текст (например, `Context: [Dynamic Label]`) в хедере или рядом с именем, чтобы юзер понимал, что ассистент видит его экран. Если это перегружает UI — оставляем "под капотом".

#### Блок 9: Архитектура взаимодействия (Универсальный Виджет) и Кастомизация
**Задача:** Дизайн должен учитывать, что виджет не содержит хардкода конкретного приложения и всецело полагается на внешнюю конфигурацию (WidgetConfig).
**Элементы и требования:**
* **Разделение режимов (Client vs Product):** На этапе создания/конфигурации виджета (в интерфейсе CMS) должен быть предусмотрен выбор режима работы (например, через тогглы):
  * **Виджет для клиентских сайтов:** Максимально облегченная версия. Работает через базовые триггеры и экшены. Для сложной навигации требует интеграции со стороны клиента (обработка Webhooks).
  * **Виджет для продуктовых сайтов (Pitch Avatar):** Включает продуктово-специфичные экшены, внутреннюю навигацию, запуск туров (Stonly) и т.д.
* **Кастомизация UI (WidgetConfig):** Технологически предусмотрена настройка UI виджета под нужды любого сайта через конфигурацию: скрытие FAB (`hideFab`), изменение имени бота, цвета кнопок, отключение лишних компонентов (например, Presentation Slides убраны для облегчения бандла).
* **OUTBOUND (Аватар -> Host App):** Логика действий аватара не хардкодится. Клик по кнопке действия (Inline Action Button) отправляет событие (Outgoing Webhook / Event) в Host App. Host App ловит это событие и выполняет бизнес-логику (открывает окно, запускает тур).
* **INBOUND API (Host App -> Аватар):** Host App передает виджету контекст (например, текущую страницу для RAG) и системные события. Виджет реагирует на них отображением бабблов (Proactive Bubbles) или сменой стейта.

### Definition of Done для Дизайна

* ✅ Отрисован общий layout виджета **согласно эталону** (логотип в хедере, видео-аватар, бабблы с outline у бота), с учетом возможности кастомизации брендинга (WidgetConfig).
* ✅ Sound Toggle расположен **в зоне аватара** (правый верхний угол), имя аватара — в **левом нижнем углу** зоны видео.
* ✅ Показаны состояния Avatar Area (`idle`, `thinking`, `speaking`).
* ✅ Бабблы бота — outline-стиль (цвет рамки кастомизируется). Бабблы пользователя — белый/серый фон без акцентного цвета.
* ✅ Отрисованы **интерактивные элементы** внутри бабблов бота: обычные текстовые ссылки и **кнопки действий (Action Buttons)** (генерирующие Outbound Webhooks для Host App).
* ✅ Input Panel: поле без рамки + акцентная круглая кнопка отправки + иконка микрофона.
* ✅ Состояния микрофона: обычное + активный Continuous Voice.
* ✅ Отрисован дизайн Proactive Bubbles поверх свёрнутого FAB, вызываемых через Inbound API.
* ✅ Отрисовано **Модальное окно** с кнопками.
* ✅ Макеты финализированы в Figma, компоненты и стили организованы, готовы к передаче в разработку.
