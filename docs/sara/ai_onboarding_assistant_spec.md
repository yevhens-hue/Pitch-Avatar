## 🎯 Мета функціоналу

Sara AI Onboarding Assistant — це проактивний AI-асистент, вбудований у продукт, який:

1. **Бачить контекст сторінки** — отримує через REST API від Host App інформацію про поточну сторінку, проект, помилки.
2. **Є повністю незалежним модулем** — не містить жорстко закодованої бізнес-логіки конкретного продукту, а вся поведінка керується універсальними тригерами.
3. **Проактивно пропонує допомогу** — на основі переданих через API подій та налаштованих тригерів.
4. **Відповідає на запитання** — на основі контексту поточної сторінки та бази знань.
5. **Виконує дії** — повідомляє Host App через Outbound API/Events про необхідність відкрити вікно чи запустити сценарій.

## 🏗️ Архітектура

### Схема комунікації (Нова Парадигма)

```
Host App (Pitch Avatar / Web App)
    │
    ├── REST API (Inbound) ────────► Sara Widget (Independent Module)
    │   (POST /context, /event)         │
    │                                   ├── Trigger Engine (No hardcoded logic)
    │                                   └── LLM Communication
    │
    ◄── Outbound API (Events/Webhooks) ─
        (trigger UX, open windows)
```

| Метод | Напрям | Призначення |
|-------|--------|-------------|
| **REST API (Inbound)** | Host App → Widget | Передача контексту (URL, проект), подій та параметрів налаштування UI. |
| **Outbound Events** | Widget → Host App | Зворотний зв'язок (через postMessage або Callback), команди на відкриття вікон UX. |
| **LLM API** | Widget ↔ Backend | Спілкування з ШІ-моделлю. |

### Система тригерів (Trigger Engine)

- Уся поведінка Widget повинна налаштовуватися через **Trigger-и**. Нова бізнес-логіка не повинна програмуватися безпосередньо у Widget.
- Тригери запускаються на основі **подій, переданих через REST API**.
- Тригери визначають: реакцію AI, відкриття сторінок, запуск сценаріїв (через Outbound команди).

## 📊 Система промтів (Prompt Architecture)

### Поточні проблеми
- Промти формуються відразу з частин на боці **PHP** → погана підтримуваність.
- Великі контексти можуть викликати **галюцинації LLM**.

### Рішення (Ідея Костянтина: зовнішнє сховище параметрів)
- Перенести формування промтів на сторону **Go-сервісу**.
- **Не закидати весь контекст сторінки відразу в LLM.**
- Описи сторінок (контекст від Надії) зберігаються в **зовнішній Knowledge Base / Storage** як параметри (наприклад, параметр `page_context_<url>`).
- При зміні сторінки система через тригер підвантажує лише той шматок бази знань (текстовий параметр), який асоційований з поточним URL.
- Це дозволить уникнути галюцинацій та оптимізувати розмір промта.

## 📋 Контекст сторінки (Page Context)

### Що передається в LLM з кожним запитом

Контекст більше не збирається самим віджетом, а **пушиться в нього через REST API** з боку Host App.

| Поле | Опис | Джерело |
|------|------|---------|
| `currentUrl` / `currentPage` | Поточна сторінка / URL | Host App REST API |
| `projectContext` | Дані про поточний проект | Host App REST API |
| `errors` / `warnings` | Поточні помилки / події | Host App REST API |
| `pageDescription` | Текстовий опис сторінки | Knowledge Base (на основі URL) |

### Як передається
- Веб-застосунок (Host App) відстежує стан і викликає API віджета.

## 🔗 Інтеграція з зовнішніми додатками (наприклад, AutoCAD)

### Що потрібно надати клієнтам
1. **API** для навігації та показу вікон у Pitch Avatar.
2. **Унікальні ID** для кожного вікна/елемента UI.
3. Спосіб **передавати контекст** поточної сторінки та помилки від їх додатку.
4. Механізм **отримувати команди** від асистента Sara.

### Поточна реалізація навігації
- Навігація через **URL-параметри**.
- `OpenURL` action з параметром `useNavigate`.

## 💡 Ключові принципи (прийняті на зустрічі)

> **Фокус на простоті:** MVP повинен бути простим і швидким для реалізації.

> **Переиспользование існуючого коду:** Багато функціональності вже реалізовано в системі тригерів — використовуємо її.

> **Контекст — ключ до якості:** Правильний контекст сторінки забезпечує якісні відповіді LLM. Без контексту — галюцинації.

> **Поступовий розвиток:** Починаємо з простого (навігація + текст), потім нарощуємо функціональність.
