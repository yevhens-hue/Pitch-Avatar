## 🎯 Мета функціоналу

Sara AI Onboarding Assistant — це проактивний AI-асистент, вбудований у продукт, який:

1. **Бачить контекст сторінки** — знає, на якій сторінці/URL знаходиться користувач, активну вкладку, помилки та попередження.
2. **Проактивно пропонує допомогу** — без запиту з боку користувача бачить дії користувача та може запропонувати підказку.
3. **Відповідає на запитання** — відповідає на запитання користувачів на основі контексту поточної сторінки та бази знань.
4. **Виконує дії** — відкриває вікна, діалоги, модальні вікна та здійснює навігацію в додатку на основі тригерів.
5. **НЕ вводить дані** — асистент тільки показує UI-елементи та навігує, не заповнює форми.

## 🏗️ Архітектура

### Схема комунікації

```
Frontend (Next.js)
    │
    ├── WebSocket ←─── Backend (Go-сервіс) ──── LLM (OpenAI/etc.)
    │   (events: Backend → Frontend)                │
    │                                               │
    └── REST API ────────────────────────────────► │
        (двостороння комунікація)
```

| Метод | Напрям | Призначення |
|-------|--------|-------------|
| **WebSocket** | Backend ↔ Frontend | Push-події від бекенду до фронтенду (команди асистента) |
| **REST API** | Frontend ↔ Backend | Двостороння комунікація: запити, відповіді |
| **postMessage API** | Host App → Widget | Фронтенд-комунікація: передача помилок та подій з CMS (хост-додатка) безпосередньо у віджет |

### Система тригерів

- Тригери запускаються на основі **подій на сторінці**.
- Кожен тригер асоціюється з **дією** (відкрити вікно, показати відео, змінити сторінку).
- **База знань** описує, що можна робити на кожній сторінці.

## 📊 Система промтів (Prompt Architecture)

### Поточні проблеми
- Промти формуються відразу з частин на боці **PHP** → погана підтримуваність.
- Великі контексти можуть викликати **галюцинації LLM**.

### Рішення (Ідея Костянтина: зовнішнє сховище параметрів)
- Перенести формування промтів на сторону **Go-сервісу**.
- **Не закидати весь контекст сторінки відразу в LLM.**
- Описи сторінок (контекст від Надії) зберігаються в **зовнішній Knowledge Base / Storage** як параметри (наприклад, параметр `page_context_<url>`).
- При зміні сторінки система через тригер підвантажує лише той шматок бази знань (текстовий параметр), який асоційований з поточним URL.
- Це дозволить уникнути галюцинацій та оптимізувати розмір промта.

## 📋 Контекст сторінки (Page Context)

### Що передається в LLM з кожним запитом

| Поле | Опис | Відповідальний |
|------|------|---------------|
| `currentUrl` / `currentPage` | Поточна сторінка / URL | Юрій |
| `activeTab` | Активна вкладка (якщо є) | Юрій |
| `errors` | Поточні помилки | Юрій |
| `warnings` | Поточні попередження | Юрій |
| `pageDescription` | Текстовий опис сторінки та що на ній можна робити | **Надія** |

### Як передається
- Через **параметри системи** (system parameters).
- Надія готує текстові описи для кожної сторінки — це ключова робота для якості відповідей LLM.

---

## 🔗 Інтеграція з зовнішніми додатками (наприклад, AutoCAD)

### Що потрібно надати клієнтам
1. **API** для навігації та показу вікон у Pitch Avatar.
2. **Унікальні ID** для кожного вікна/елемента UI.
3. Спосіб **передавати контекст** поточної сторінки та помилки від їх додатку.
4. Механізм **отримувати команди** від асистента Sara.

### Поточна реалізація навігації
- Навігація через **URL-параметри**.
- `OpenURL` action з параметром `useNavigate`.

## 💡 Ключові принципи (прийняті на зустрічі)

> **Фокус на простоті:** MVP повинен бути простим і швидким для реалізації.

> **Переиспользование існуючого коду:** Багато функціональності вже реалізовано в системі тригерів — використовуємо її.

> **Контекст — ключ до якості:** Правильний контекст сторінки забезпечує якісні відповіді LLM. Без контексту — галюцинації.

> **Поступовий розвиток:** Починаємо з простого (навігація + текст), потім нарощуємо функціональність.
