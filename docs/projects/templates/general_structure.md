# Архитектура и общая структура: Шаблоны как нативные объекты
**Роль:** AI CTO
**Проект:** Pitch Avatar
**Дата:** Май 2026

## 1. Цель системы
Переход системы шаблонов Pitch Avatar от статичных, предварительно отрендеренных файлов PPTX к динамическим нативным структурам данных (слайды на основе JSON). Этот сдвиг обеспечивает мгновенный перевод, редактирование в реальном времени и динамическую генерацию сценариев для аватаров.

## 2. Архитектура модели данных (нативные объекты)

Для реализации видения CEO каждый «Слайд» внутри шаблона должен быть деконструирован в многослойную объектную модель.

### 2.1 Схема слайда (концептуальная)
```json
{
  "slideId": "uuid",
  "templateId": "uuid",
  "background": {
    "type": "image | color",
    "value": "#FFFFFF или https://storage.../bg.jpg"
  },
  "objects": [
    {
      "objectId": "uuid",
      "type": "text | image",
      "zIndex": 1,
      "content": "Добро пожаловать в Pitch Avatar",
      "styles": { "font": "Inter", "size": "24px", "color": "#000", "x": 100, "y": 200 }
    },
    {
      "objectId": "uuid",
      "type": "image",
      "zIndex": 2,
      "src": "https://storage.../logo.png",
      "styles": { "width": "50px", "height": "50px", "x": 10, "y": 10 }
    }
  ],
  "avatarScript": "Здравствуйте и добро пожаловать на эту демонстрацию..."
}
```

## 3. Конвейер импорта PPTX на базе LLM

Основная техническая задача (Задача 2 от CEO) — это «Импорт PPTX в объекты с использованием LLM с ограничениями». Поскольку нативные библиотеки парсинга PPTX (такие как `pptx` на python или `officegen` в Node) с трудом справляются со сложным форматированием, мы будем использовать гибридный подход.

### Рабочий процесс:
1.  **Загрузка PPTX:** Пользователь или администратор загружает файл шаблона PPTX.
2.  **Извлечение визуальной части (Фон):** Система рендерит слайд *без* текста для извлечения базового фонового изображения (Задача 1 от CEO).
3.  **Извлечение данных (LLM / OCR):** 
    *   Извлечение необработанного текста и ограничивающих рамок (bounding boxes) из XML-файла PPTX.
    *   Передача изображения слайда и текста из XML в мультимодальную LLM (например, GPT-4o или Claude 3.5 Sonnet).
    *   **Инструкция промпта:** «Проанализируй этот слайд. Верни JSON-массив всех текстовых объектов, включая их относительные координаты X/Y, примерный размер шрифта и содержание текста. Игнорируй графику фона».
4.  **Реконструкция:** Бэкенд сопоставляет JSON-вывод LLM с объектами `Slide Schema` Pitch Avatar.
5.  **Применение ограничений:** Если LLM идентифицирует сложные объекты (диаграммы, SmartArt, сгруппированные SVG), система рендерит эту конкретную область как статичный графический объект для сохранения визуальной точности, упрощая дизайн до приемлемого для шаблонов уровня.

## 4. Управление Z-Index (порядок рендеринга)

(Задача 3 от CEO): «Порядок отображения объектов на слайде».
*   В базе данных массив `objects` будет иметь целочисленное свойство `zIndex`.
*   При рендеринге в DOM (React/Vue) эти объекты будут сопоставлены с абсолютно позиционированными элементами `<div>` с `z-index: ${object.zIndex}`.
*   Конвейер импорта попытается определить z-index на основе порядка элементов в XML-файле PPTX (рендеринг снизу вверх).

## 5. Безопасность и масштабируемость
*   **Хранение:** Фоны и извлеченные изображения будут храниться в S3/Cloud Storage со ссылками по URL.
*   **Производительность:** Загрузка шаблона означает получение одного JSON-файла и рендеринг объектов DOM, что значительно быстрее и менее ресурсозатратно, чем потоковая передача полного видео или передача тяжелых PDF-файлов.
