# Эволюция системы шаблонов в Pitch Avatar

Этот документ описывает архитектурный переход от старой системы шаблонов (генерация и копирование `.pptx`) к новой нативной системе (JSON-проекты и Constrained UI). Документ консолидирует требования CEO, старую техническую документацию и актуальное состояние системы.

## 1. Бизнес-видение и ценность (Требования CEO)
Главная цель новой системы шаблонов — сократить путь пользователя до **"Aha-момента"** (момента осознания ценности продукта). 
Вместо того чтобы собирать презентацию и настраивать аватара с нуля, система дает готовые B2B Use Cases (HR Assistant, Product Demo, Compliance Training и т.д.).

### User Stories
* **Презентатор (Пользователь):** Хочет видеть готовые примеры, смотреть их превью *без открытия самого проекта*, и по кнопке "Use Template" получать полную копию для быстрого редактирования. Это избавляет от проблемы "чистого листа".
* **Администратор:** Хочет создавать шаблоны, привязывать их к эталонным проектам, управлять их порядком на Главной странице (Home Page) или скрывать.

### Реализация в UI (клиентская часть)
* **Home Page:** Шаблоны выводятся в горизонтальной скроллируемой ленте под блоком "Project Wizards" (до 4 шаблонов на экране).
* **Сайдбар (Templates):** Отдельная галерея шаблонов сеткой с пагинацией. Есть поиск и фильтрация по типу (`All types`).
* **Модальное превью:** Карточка шаблона содержит превью слайдов, описание и кнопку "Use Template". При ее нажатии происходит копирование скрытого базового проекта в рабочую среду пользователя.

## 2. Старая архитектура (Legacy API и CopyService)
Раньше шаблоны физически ничем не отличались от обычных презентаций, кроме флага `is_template` в базе данных.

### Разделение шаблонов (is_simple_template)
Система жестко делила шаблоны на два типа через флаги и API-фильтры:
1. **Шаблоны для копирования (`is_simple_template = true`)**: Предназначались для простого дублирования.
2. **Шаблоны для генерации (`is_simple_template = false`)**: Предназначались для генерации AI-презентаций по title или контенту.

### Эндпоинт создания (`api/cms/presentations/create-from-template`)
* Для копирования бэкенд использовал тяжелый скрипт **CopyService**, который рекурсивно дублировал все связанные записи в реляционной БД.
* Аргумент `$overrides` позволял подменять параметры "на лету" (например, title или owner_id).
* Существовал захардкоженный шаблон `Blank` (белый фон), который всегда выводился первым.

### Микро-генерация медиа (presentation-step-controls)
Шаблоны содержали зашитые JSON:API контролы. Например, контрол с типом `image-output` мог содержать настройки `control_settings: { "image_generate": true, "image_prompt": "happy presenter listen music" }`. 
Во время работы CopyService эти триггеры ловились бэкендом, запускалась генерация картинки через нейросеть (Stable Diffusion / Midjourney API), и готовая картинка подставлялась в скопированную презентацию. Также поддерживалась генерация текстов, голосов (Azure, ElevenLabs, Polly) и видео от D-ID.
Администраторская панель для старой системы позволяла загружать файлы `.pptx` напрямую, разбивая их на слои, но не поддерживала `.pptx` анимации.

## 3. Новая архитектура (Native JSON и Source Project)
Новый подход ушел от "черного ящика" тяжелой серверной генерации в пользу прозрачных эталонных проектов и визуального редактора на фронтенде.

### Суть "Обертки" (Source Project)
В новой парадигме **Шаблон — это только обертка/метаданные** (Название, Описание, Иконки, Приоритет отображения). Вся бизнес-логика шаблона лежит в привязанном к нему **Source Project** (`selected_project_id`). Это обычный проект Pitch Avatar, к которому есть доступ только у админа.

Когда пользователь нажимает "Use Template":
1. Фронтенд или быстрый слой бэкенда мгновенно копирует JSON-структуру `Source Project`.
2. Пользователь попадает в редактор, где уже настроены слайды, тексты и ИИ-ассистент.

### Constrained UI (Ограниченное редактирование)
Пользователь не генерирует презентацию с нуля, он работает в рамках заданного дизайна:
* Фоновые картинки и структура (layout) заблокированы, чтобы предотвратить поломку дизайна.
* **Translate & Voice:** Так как текст теперь хранится в нативном JSON, пользователь может в один клик локализовать все текстовые слои слайда и скрипт аватара на 30+ языков прямо в браузере. Эта логика заменила серверную генерацию на лету.

### Новая Панель Администратора (CRUD)
В отдельном приложении для админов (или разделе админки) реализован CRUD:
* **Поля шаблона:** Thumbnail, Name, Description, Source Project (выбор из списка готовых проектов), Status (Active/Inactive), Home Order (порядок на главной).
* Больше не нужно загружать `.pptx` с триггерами `image_generate`. Админ просто собирает эталонный проект во встроенном редакторе Pitch Avatar и привязывает его к шаблону.

## 4. Сравнительная таблица (Old vs New)

| Параметр | Старая архитектура (Legacy API) | Новая архитектура (Native JSON) |
| :--- | :--- | :--- |
| **Структура хранения** | Реляционная таблица `is_template`, тяжелые связи. | Легковесная таблица шаблонов-оберток (Template -> Source Project). |
| **Процесс копирования** | Серверный скрипт `CopyService` (длительная операция). | Мгновенное дублирование JSON-скелета проекта. |
| **Основа шаблона** | Загруженный `.pptx` и скрипты. | Нативный проект, собранный в редакторе Pitch Avatar. |
| **Генерация контента (ИИ)** | На бэкенде в процессе копирования (по триггерам `image_generate`). | Интерактивно во фронтенде (Constrained UI, кнопка Translate & Voice). |
| **Разделение типов** | Разделение по API-флагам `for_copy` / `for_ai_generation`. | Единая концепция `Source Project`. Дополнительные генерации по промпту вынесены в отдельные флоу (Wizards). |
