# Unauthenticated User Impersonation in /api/sara/create-project

**ID:** vuln-0001
**Severity:** HIGH
**Found:** 2026-07-18 19:35:57 UTC
**Target:** https://github.com/Pitch-Avatar/src — `/workspace/src/app/api/sara/create-project/route.ts`
**Endpoint:** /api/sara/create-project
**Method:** POST
**CWE:** CWE-639
**CVSS:** 7.5
**Fix Effort:** Low

## Description

The `POST /api/sara/create-project` API endpoint accepts a `userId` field from the unauthenticated request body and creates a database record attributed to that user, with no verification that the caller is actually authenticated as that user. Because the handler uses the Supabase **service role key** (which bypasses all Row Level Security policies), any anonymous internet caller can create projects owned by an arbitrary Supabase user UUID.

## Evidence

Full route handler (lines 18–79) — no authentication check exists anywhere before the database insert:

```typescript title=app/api/sara/create-project/route.ts startLineNumber=18 endLineNumber=54
export async function POST(req: Request) {
  try {
    const body = await req.json();
    const { type, title, avatarName, role, userId } = body;   // userId from attacker

    console.log('[Sara CreateProject] called:', { type, userId, hasServiceKey: !!SERVICE_ROLE_KEY });

    if (!type) {
      return NextResponse.json({ error: 'type is required' }, { status: 400 });
    }
    if (!userId) {
      return NextResponse.json({ error: 'userId is required' }, { status: 400 });
    }

    const key = SERVICE_ROLE_KEY || ANON_KEY;   // service key bypasses RLS
    const supabase = createClient(SUPABASE_URL, key, {
      auth: { autoRefreshToken: false, persistSession: false },
    });

    const dbType = type === 'video' ? 'video' : 'presentation';
    const projectTitle = type === 'chat-avatar'
      ? `${avatarName || 'AI Avatar'} (Chat Avatar)`
      : (title || 'New Presentation');

    const { data: project, error } = await supabase
      .from('projects')
      .insert([{
        title: projectTitle,
        type: dbType,
        status: 'active',
        user_id: userId,             // victim's UUID inserted verbatim
      }])
      .select()
      .single();
```

No `Authorization` header check, no session cookie lookup, and no call to any Supabase auth API appears anywhere in the file.

## Impact

An unauthenticated attacker who knows (or enumerates) a victim's Supabase UUID can flood the victim's project library with arbitrary projects, corrupt their workspace state, and potentially interfere with features that depend on project counts or project IDs. Because the service role key bypasses RLS, this write path has no database-level backstop. If project creation triggers downstream side-effects (webhook calls, storage allocation, billing events), the impact surface expands further.

2. Send an unauthenticated HTTP POST to `/api/sara/create-project` with a JSON body specifying the target `userId`.
3. Observe that the server responds with `200 OK`, a new `projectId`, and a `redirectUrl` — confirming the project was inserted under the victim's account.

## Remediation

Validate the caller's identity server-side before using any user identifier:

1. At the start of the handler, retrieve the authenticated session from the request cookies using Supabase's SSR helper (`createServerClient` with `@supabase/ssr`, or `createRouteHandlerClient` with `@supabase/auth-helpers-nextjs`). Return `401 Unauthorized` if no valid session exists.
2. Derive `userId` exclusively from `session.user.id` — never accept it as a body parameter. If a `userId` field is still included in the body for legacy compatibility, reject the request with `403 Forbidden` when it does not match the session user's ID.
3. Remove the service role key from this route. Use a session-scoped Supabase client so that RLS policies enforce ownership at the database layer as a defence-in-depth measure.
4. Remove the hardcoded fallback value for `ANON_KEY` on line 6. Require the environment variable to be explicitly set; fail fast at startup if it is absent.
5. Rotate the exposed anon key (`sb_publishable_juNeZXupS_SXWtvvK1MdLw_gVRnqhsE`) immediately, as it is now permanently committed to source history.

## Assumptions

The attacker can reach the endpoint over the internet. Knowledge of a victim's Supabase UUID is required but can often be obtained from public profile pages, API responses that leak user IDs, or by brute-forcing UUID ranges. No credentials of any kind are required to exploit this endpoint.
